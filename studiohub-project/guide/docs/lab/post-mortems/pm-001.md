# 🔬 PM-001 — Índice Inexistente em Produção

**Incidente relacionado:** [Incident #014](/lab/incidents/incident-014)  
**Data:** 15 Jun 2026  
**Status:** ✅ Publicado  
**Autores:** Diego

---

<div class="pm-section">
<div class="pm-question">O que aconteceu?</div>
<div class="pm-answer">

O endpoint `/api/appointments` ficou com latência de **12–18 segundos** durante ~3 horas em produção, afetando aproximadamente 340 clientes. A causa foi a ausência de índice composto em `(studio_id, date)` na tabela `appointments`, que cresceu para 1.2M de registros.

</div>
</div>

<div class="pm-section">
<div class="pm-question">Como descobrimos?</div>
<div class="pm-answer">

Alerta do **Grafana** disparou automaticamente às 14h37 com p95 de latência acima de 10 segundos. Sem o monitoramento, o incidente poderia durar horas a mais. A observabilidade implementada na **Sprint 03** foi crucial.

</div>
</div>

<div class="pm-section">
<div class="pm-question">Por que aconteceu?</div>
<div class="pm-answer">

**Causa raiz:** ausência de processo formal de revisão de performance antes de deploy em produção.

**Causas contribuintes:**

1. `EXPLAIN ANALYZE` foi feito apenas em ambiente de desenvolvimento com ~500 registros
2. Não existia checklist de "impacto em tabelas com alto volume"
3. A feature foi desenvolvida quando a tabela ainda era pequena (< 10k registros)
4. Não havia query slowlog configurado para detectar degradação gradual

</div>
</div>

<div class="pm-section">
<div class="pm-question">Como foi resolvido?</div>
<div class="pm-answer">

1. Identificação via trace distribuído no Datadog APM (15min de investigação)
2. Criação de migration com `CREATE INDEX CONCURRENTLY` (sem downtime)
3. Deploy em produção às 16h30
4. Monitoramento por 1h após deploy para confirmar estabilização
5. Incidente encerrado às 17h49

</div>
</div>

<div class="pm-section">
<div class="pm-question">Como evitar no futuro?</div>
<div class="pm-answer">

**Mudanças de processo:**

- [ ] Checklist de PR: "Essa query funciona bem com 1M+ registros?"
- [ ] `EXPLAIN ANALYZE` obrigatório para queries em tabelas críticas
- [ ] Alerta de query slowlog no Datadog (threshold: 500ms)

**Mudanças de código:**

- [ ] Adicionar `Index` explícito no schema Prisma para todas as foreign keys críticas
- [ ] Script de análise de índices faltando no CI/CD

</div>
</div>

<div class="pm-section">
<div class="pm-question">Quais ADRs foram gerados ou modificados?</div>
<div class="pm-answer">

- ✅ **ADR-019:** Política de índices em tabelas de alto volume (novo)
- ✅ **ADR-020:** Query analysis obrigatória antes de deploy em tabelas > 100k registros (novo)

</div>
</div>

<div class="pm-section">
<div class="pm-question">Quais testes foram adicionados?</div>
<div class="pm-answer">

- ✅ Teste de performance com seed de 500k registros no ambiente de staging
- ✅ Teste de integração verificando que o índice existe após migrations
- ✅ Benchmark de latência p95 no endpoint `/appointments` (limite: 300ms)

</div>
</div>

## Aprendizado Pessoal

> "Esse incidente me ensinou que **volumetria de dados muda completamente o comportamento de uma query**. O que funciona com 500 registros pode ser catastrófico com 1.2M. Desde então, sempre testo com dados realistas. E o `EXPLAIN ANALYZE` virou ritual sagrado." 😂
